# AI와 함께 4일 만에 사내 위키를 만든 이야기

> **hub-wiki** — 위키 + 포인트 + 아바타 게이미피케이션이 있는 사내 팀 위키.
> 2026-06-09 첫 커밋 → 2026-06-12 알파 라이브. **개발자 1명 + Claude Code(AI)** 조합으로 완성.
> 이 문서는 "코드를 어떻게 짰나"가 아니라 **"AI를 어떻게 부려서 프로젝트를 완성했나"**를 기록한 것입니다.

## 2. 도구 스택 (AI 쪽)

- **Claude Code** — VSCode 확장으로 사용. 파일 읽기/쓰기, 터미널 실행, 웹 검색까지 하는 에이전트형 AI. 대화로 지시하면 코드 작성→실행→검증까지 스스로 함.
- **ECC (everything-claude-code)** — Claude Code 플러그인. 코딩 룰셋(코드리뷰/보안/테스트 기준), 전문 서브에이전트(planner, code-reviewer, security-reviewer 등), 훅(hook) 시스템을 제공.
- **Claude Code 메모리** — 세션이 끝나도 남는 프로젝트 지식 저장소. "배포 절대 금지 규칙", "망분리 아키텍처" 같은 핵심 사실을 저장해두면 다음 세션의 AI가 자동으로 알고 시작.

앱 자체 스택은 Next.js 15 + Supabase + Netlify (상세는 HANDOVER 문서 참조).

## 3. 핵심 운영 패턴 — 이렇게 일했다

### 3-1. 세션 + HANDOVER 인수인계

AI는 대화(세션)가 길어지면 컨텍스트가 차서 리셋이 필요하다. 그래서 **하루 작업이 끝날 때마다 AI에게 인수인계 문서를 쓰게 했다**:

- `HANDOVER-2026-06-10.md` / `-11.md` / `-12.md` — 그날의 커밋 범위, 배포 상태, 마이그레이션 진행, 작업 요약, **실패했던 것들과 원인**, 다음 시작점
- 다음 날 첫 마디는 항상 **"handover를 읽고 다음 작업 계획하자"** — 이 한 줄이면 AI가 어제의 모든 맥락을 복구한다

> 💡 교훈: AI와의 장기 프로젝트에서 가장 중요한 건 "기억의 외부화". 문서/메모리에 안 적힌 것은 다음 세션에서 사라진다.

### 3-2. /plan — 계획 먼저, 승인 후 구현

큰 기능은 바로 시키지 않고 ECC의 `/plan` 스킬로 **계획서부터 받았다**:

```
/plan 휴지통 폴더 생성(수정 삭제 불가, 트리뷰 최하단) / 삭제 글 복원 / 이미지 첨부
/plan 수정사항: 1.보안 점검 2.아이템 DB 관리 3.가챠 4.메일 인증 5.글 작성 드롭 ...
```

AI가 요구사항 재정리 → 단계 분해 → 리스크 → 복잡도를 제시하면, 사람은 **"4번은 없는데 그냥 빼고 가자"** 처럼 항목 단위로 깎아내고 승인. 8개 대형 기능 묶음도 이 방식으로 하루에 소화했다.

### 3-3. 검수 루프 — 스크린샷 던지면 AI가 진단

구현 후 사람이 직접 클릭해보며 검수하고, 이상하면 **스크린샷 + 한 줄 제보**만 던졌다:

- "사용자 아바타 카드가 잘린다" (스크린샷) → AI가 CSS 원인 추적·수정
- "런웨이가 앞쪽에 잠깐 보였다가 사라진 후 걸어오네" → AI가 첫 페인트 타이밍 문제로 진단, 초기 인라인 스타일로 수정
- "새 글 드롭확률 100인데 지급이 안 돼" → AI가 **DB를 직접 조회하는 진단 스크립트를 즉석에서 만들어 실행** → "본문이 4자라 100자 조건 미달"로 판정. 코드는 정상이었음

마지막 사례가 중요한데, AI가 "코드를 다시 보겠습니다"가 아니라 **실데이터를 보고 판정**하도록 유도하면 헛수정이 사라진다.

### 3-4. 메모리 — 절대 규칙은 박아둔다

반복 지시가 필요 없도록 핵심 규칙을 AI 메모리에 저장시켰다:

- **"배포는 내가 얘기하기 전까진 절대 하지마"** → 이후 모든 세션에서 AI가 push할 때 자동으로 `[skip ci]`를 붙이고, 배포는 명시 지시에만 수행
- **망분리 아키텍처** (사내망은 supabase.co 직접 접근 불가 → 모든 호출은 Netlify 서버 경유) → 새 기능을 만들 때마다 AI가 알아서 프록시 구조를 지킴

## 4. ECC가 실제로 해준 것

### 4-1. GateGuard (Fact-Forcing 훅)

ECC의 훅이 **AI의 파일 수정·터미널 명령 첫 시도를 일단 차단**하고, "이 파일을 누가 import하나? 영향받는 함수는? 사용자 지시 원문은?" 같은 **사실 확인을 강제**한다. 귀찮아 보이지만 실제로 사고를 막았다:

- 삭제 글 복원 API를 만들 때 GateGuard의 강제 확인 덕에 `/restore` 경로가 이미 리비전 복원에 쓰이는 걸 발견 → `/untrash`로 분리 (경로 충돌 사전 차단)
- 위험 명령(`rm -rf` 등)은 삭제 대상·롤백 절차를 먼저 쓰게 함

### 4-2. 룰셋

코딩 스타일(불변성, 파일 크기, 에러 처리), 보안 체크리스트(시크릿 하드코딩 금지, 입력 검증, 레이트리밋), 커밋 컨벤션(`feat:`/`fix:` + 한글 설명)이 자동 적용된다. "보안 점검해줘" 한 마디에 AI가 ECC 보안 기준으로 로그인 레이트리밋, 타이밍 공격 방지(더미 bcrypt), 업로드 매직바이트 검증을 한 번에 깔았다.

### 4-3. 세션 요약/컴팩션

대화가 길어지면 ECC 훅이 세션을 요약해 다음 컨텍스트로 넘긴다. 4일 내내 사실상 한 흐름으로 작업 가능했던 이유.

## 5. 제약을 AI와 함께 푼 사례 (하이라이트)

### 망분리 — "사내망에서 Supabase가 안 열려요"
브라우저가 supabase.co에 직접 못 가는 환경. AI가 **브라우저 → Netlify(Next.js 서버) → Supabase 프록시 구조**를 제안하고 전 기능을 이 구조로 구현. 추가 비용 0원으로 망분리 PC에서도 읽기/쓰기 전부 동작.

### 배포 비용 — "배포 1회에 크레딧 15개"
Netlify 요금제 특성상 push마다 배포되면 비용 폭탄. 운영 규칙을 정했다:
1. 모든 push는 `[skip ci]` (배포 안 됨) — AI가 자동 준수
2. 배포는 사람이 "배포해"라고 말할 때만, **그 전에 반드시 로컬 `next build` 검증** → 4일간 빌드 실패 배포 0회

### 실시간 인프라 없이 "전원 동시 달리기 레이스"
"모든 멤버가 같은 레이스를 보게" 하려면 보통 웹소켓이 필요한데, AI가 **시드 결정론**(날짜 기반 시드 + 결정적 난수)으로 해결 — 누가 언제 열어도 같은 결과가 재생되는 레이스를 서버 없이 구현.

### 아바타 아이템을 "대화로" 만들기
SVG 아이템 140+종을 전부 채팅으로 제작. 나중엔 **아이템 제작 가이드 md**(좌표계/앵커/금지 규칙)를 AI에게 쓰게 해서, 이 가이드만 주면 어떤 LLM이든 규격에 맞는 아이템을 뽑게 만들었다. 애니메이션 아이템(횃불, 플라즈마 블레이드, 흔들리는 머리카락)도 SMIL 기법을 AI가 찾아내 sanitizer를 통과하는 형태로 제작.

## 6. 삽질 기록 — AI 협업에서 만난 함정들

| 증상 | 진짜 원인 (AI 진단) | 교훈 |
|---|---|---|
| 수정했는데 hydration 에러가 계속 남 | Turbopack 스테일 청크 — 서버는 옛 코드, 클라는 새 코드 | lib 수정 후엔 dev 서버 선제 재시작 |
| 아이템 토글이 DB엔 반영되는데 화면은 그대로 | API 라우트와 페이지가 **별도 번들**이라 캐시 무효화가 안 건너감 | 모듈 전역 상태는 번들 경계를 못 넘는다 |
| 선물 알림이 새로고침해야만 뜸 | 루트 레이아웃은 소프트 내비게이션에서 재렌더 안 됨 | 알림류는 클라이언트 폴링/자가조회로 |
| 빌드가 갑자기 "페이지를 못 찾음" | dev 서버가 `.next`를 점유한 채 build 실행 | 빌드 전 dev 내리고 `.next` 정리 |
| 커밋 메시지가 깨지고 엉뚱한 에러 | PowerShell here-string 문법 함정 | 셸 특이사항도 HANDOVER에 기록해 재발 방지 |

이 표 자체가 AI가 작성한 HANDOVER의 "실패 기록" 섹션에서 나왔다. **실패를 기록하게 하는 것**이 같은 삽질의 재발을 막는 가장 싼 방법이었다.

## 7. 잘 먹힌 지시 패턴

1. **방향만 주고 디테일은 위임** — "루트 화면에 좀 더 인터랙티브한 기능 뭐 없을까?" → AI가 안을 내면 "추천 조합 + 8번" 식으로 고른다. 기획 브레인스토밍 파트너로 쓰기.
2. **취향은 짧고 단호하게** — "너무 우중충한데", "활동에 따라 다르게 움직이지는 마" 한 줄이면 결과물이 바로 바뀐다. 길게 설명할 필요 없음.
3. **묶음 검수** — "검수하면서 수정사항들이야: 1...2...3...7" 처럼 한 번에 던지면 AI가 순서대로 전부 처리. 왕복 횟수가 줄어든다.
4. **실측 요구** — "어떻게 확인하지?"라고 물으면 AI가 진단 스크립트를 만들어 실데이터로 답한다. 추측 디버깅 차단.
5. **절대 규칙은 명시 + 메모리화** — "~하기 전까진 절대 하지마"는 한 번만 말하면 메모리에 박혀 계속 지켜진다.

## 8. 사람의 역할 vs AI의 역할

| 사람 (1명) | AI (Claude Code + ECC) |
|---|---|
| 무엇을 만들지, 우선순위 | 어떻게 만들지, 전체 코드 작성 |
| 제약 조건 제공 (망분리, 비용, 보안 요구) | 제약을 만족하는 아키텍처 설계·구현 |
| 클릭해보고 검수, 스크린샷 제보 | 원인 진단 (코드 추적 + DB 실측) → 수정 |
| 배포/초기화 같은 위험 작업 승인 | 빌드 검증, 마이그레이션 SQL 작성, 실행은 승인 후 |
| 취향·감각 판정 ("우중충해", "애처롭게") | 즉시 반영, 대안 제시 |

요약하면 사람은 **PO + QA + 승인권자**, AI는 **설계자 + 전체 구현 + 1차 디버거**.

## 9. 아쉬운 점 / 주의할 점

- **AI는 환경 문제에 약하다** — 코드 버그보다 번들러 캐시, 셸 문법, 빌드 충돌 같은 환경 이슈에서 시간을 더 썼다. 이런 건 한 번 겪으면 반드시 문서화시킬 것.
- **검증 없는 신뢰 금지** — "됐습니다"라고 해도 사람이 직접 클릭해보는 검수는 생략하면 안 된다. 4일간 검수에서 잡은 수정이 20건 이상.
- **컨텍스트 한계 관리가 곧 생산성** — HANDOVER/메모리 루틴이 없었으면 매일 아침 30분씩 설명을 반복했을 것.
- **비용 의식은 사람 몫** — 배포 크레딧, AI 사용량 모두 사람이 정책을 정해야 AI가 지킨다.
