# SVG 레이어드 아바타 시스템 — 구현 가이드 (Claude Code 핸드오프 문서)

> 이 문서는 "이미지 파일 0개로 만드는 커스터마이징 아바타 시스템"의 설계 철학과 구현 방법을
> 다른 프로젝트(팀위키의 멤버 아바타 기능)에 이식하기 위한 핸드오프 문서다.
> 구체적인 좌표 숫자는 예시일 뿐이며 그대로 쓸 필요 없다. **숫자를 정하는 "이유"와 "규칙"이 핵심이다.**

---

## 1. 핵심 개념

아바타 = **하나의 SVG 안에서 여러 파츠(레이어)를 순서대로 겹쳐 그린 것**.

- 모든 파츠(몸, 눈코입, 옷, 모자, 손 아이템…)는 PNG가 아니라 **SVG 마크업 문자열**로 데이터에 저장
- 렌더링 = 장착 상태를 보고 파츠 문자열들을 정해진 순서로 이어붙여 `svg.innerHTML`에 꽂는 것
- 따라서 **렌더러는 순수 함수**: `(얼굴 선택 + 장착 목록) → SVG 문자열`
- 이 구조의 보상: 이미지 에셋 0개, 무한 확대(벡터), 아이템 추가 = 데이터 한 줄, 애니메이션 = CSS만으로 가능

팀위키 적용 시 가장 중요한 함의: **멤버 아바타의 영속 데이터는 작은 JSON 하나**
(`{ face: {eyes, nose, mouth}, equipped: {slotId: itemId} }`)이고, 렌더 함수만 있으면
프로필 카드·댓글 옆 썸네일·멤버 목록 어디서든 같은 데이터로 아무 크기로나 그릴 수 있다.

---

## 2. 좌표 시스템 — 숫자보다 "규칙"이 중요

### 2.1 단일 viewBox 원칙 (가장 중요한 결정)

**모든 파츠는 단 하나의 공유 좌표계(viewBox) 안에서, 캐릭터가 입은 "최종 위치"에 그린다.**

레퍼런스에서는 `viewBox="0 0 320 400"`을 썼다. 이 결정의 이유:

- 세로로 긴 비율(4:5) → 전신 캐릭터 + 머리 위 모자 공간 + 발밑 바닥/장식 공간이 다 들어감
- 320×400 정도의 "정수 좌표로 손으로 그리기 편한" 크기 — SVG는 벡터라 실제 표시 크기와 무관
- **모자는 (모자 슬롯 좌표계)가 아니라 (캐릭터 머리 위 좌표)에 그린다.** 오프셋 계산, 앵커 변환,
  파츠별 좌표계 같은 게 전부 불필요해진다. 그냥 겹치면 맞는다.

이게 아이템 호환성의 전부다. **호환 = 같은 viewBox + 같은 앵커를 보고 그렸다는 약속.**

### 2.2 앵커(기준점) 문서화

캐릭터 몸을 먼저 그린 뒤, 파츠가 닿는 기준점들을 상수처럼 문서화한다. 레퍼런스 값(예시):

| 앵커 | 값 | 이것을 보고 그리는 파츠 |
|---|---|---|
| 머리 중심/반지름 | (160,120) r56 | 모자(머리 위 y≈40~85), 헤어(두피 y≈50~120), 얼굴 악세서리 |
| 왼눈/오른눈 | (140,110), (180,110) | 눈 옵션, 안경류 |
| 코 / 입 | (160,126) / (160,146) | 코·입 옵션, 수염(입 아래~턱), 마스크 |
| 몸통 박스 | x124~196, y170~268 | 상의 |
| 엉덩이~발 | y250~347, 다리 x138-154 / x166-182 | 하의 |
| 발 중심 | (146,350), (174,350) | 신발 |
| 손 중심 | 좌(103,252), 우(217,252) | 손 아이템 (여기서 위/옆으로 뻗어나가게) |
| 바닥선 | y=360 | 바닥 장식(좌 x≈55, 우 x≈265), 배경의 지면 |
| 관절 피벗 | 어깨 (135,196)/(185,196), 엉덩이 (146,292)/(174,292) | 애니메이션 회전 중심 |

새 프로젝트에서는 몸 디자인에 맞게 값을 새로 정하되, **이 표를 반드시 만들어라.**
아이템 100개를 그리는 작업은 결국 "이 표를 보고 반복하는 일"이고, 표가 없으면 파츠가 어긋난다.

### 2.3 슬롯별 미리보기 crop

아이템 카드 썸네일은 아이템 svg를 **전체 viewBox가 아니라 슬롯별 crop 영역으로 자른 작은 svg**에 넣는다:

```js
{ id: "hat", name: "모자", zoom: "88 6 144 110" }   // zoom = 썸네일용 viewBox
// 카드: <svg viewBox="${slot.zoom}">${고스트 몸체(opacity 0.22)}${item.svg}</svg>
```

- 아이템이 월드 좌표에 그려져 있으므로, viewBox만 좁히면 썸네일이 공짜로 나온다
- 착용류 파츠는 **반투명 몸체(고스트)를 밑에 깔아** 어디에 입는 건지 보여주면 UX가 확 좋아진다

## 3. 데이터 모델

### 3.1 슬롯과 아이템

```js
const SLOTS = [ { id, name, zoom }, ... ];
// 레퍼런스는 12종: 헤어/모자/얼굴악세서리/수염/상의/하의/신발/왼손/오른손/좌측바닥장식/우측바닥장식/배경
const ITEMS = { [slotId]: [ { id, name, price, svg, rigid? }, ... ] };
const FACE_OPTIONS = { eyes: [...], nose: [...], mouth: [...] };  // 무료 — 정체성은 무료, 꾸미기는 재화
```

팀위키라면 price/상점은 빼고 전부 무료 선택으로 단순화해도 됨. 슬롯 구성도 자유 —
다만 **배경 슬롯**은 프로필 카드 개성에 효과가 크니 추천.

### 3.2 아이템 svg 조각 작성 규칙 (어기면 실제로 깨지는 것들)

1. **`id` 속성 금지.** 아이템 svg는 메인 무대 + 카드 썸네일에 동시에 복제 렌더링된다.
   같은 문서에 id가 중복되면 참조가 꼬인다. → gradient/`<defs>`도 쓰지 말고 단색 + opacity로 해결
2. **클래스는 "기능 마커"로만 사용:**
   - `class="pupil"` — 눈동자 그룹. JS가 transform을 줘서 마우스를 따라가게 함
   - `class="sleeve-l"` / `"sleeve-r"` — 소매 그룹. 렌더러가 추출해 팔 스윙 그룹으로 옮김 (5.4절)
3. XML 유효성 — 자동 검증 스크립트를 만들어 돌려라 (Node로 `ITEMS` 순회 → 각 svg를 XML 파서에 넣기).
   레퍼런스에서 실제로 오타 1건을 이 검증으로 잡았다
4. 색상 팔레트를 8~12개로 제한하면 아이템 149개를 만들어도 톤이 통일된다
5. 반복 형태는 헬퍼 함수로: `topBase(color)`(몸통+소매), `pantsBase(color)`, `shoeBase(color)` —
   아이템은 "베이스 + 디테일 몇 개"로 작성량이 1/3로 줄어든다

### 3.3 상태와 저장

```js
state = {
  version: 1,                  // 스키마 버전 — 불일치 시 초기화
  face: { eyes, nose, mouth },
  equipped: { [slotId]: itemId | null },
  owned: [...], coins: n,      // 경제 기능을 쓸 때만
}
```

- 로드 시 `{ ...defaultState(), ...parsed }`로 **기본값 위에 병합** — 슬롯이 추가돼도 옛 저장이 안 깨짐
- 마이그레이션 교훈: 병합 후에 플래그를 검사하면 기본값이 덮어써서 영원히 실행 안 된다.
  **반드시 병합 전의 `parsed` 원본으로 판정**할 것 (실제로 겪은 버그)
- `preview`(입어보기) 상태는 **저장 대상 밖**에 둔다. 렌더 시 `preview[slot] ?? equipped[slot]`로 우선 적용

## 4. 렌더링 — 레이어 순서가 곧 스펙

아래에서 위로 (레퍼런스 순서, 이유와 함께):

```
1. 배경 (아이템 또는 기본 배경)
2. 좌/우 바닥 장식        ← 캐릭터가 그 "앞"을 지나가야 하므로 캐릭터보다 아래
3. 바닥 그림자 (ellipse)
4. ── #char 그룹 시작 (이 안만 움직인다) ──
   5. 왼다리/오른다리 그룹 (다리 피부 + clip된 하의 + clip된 신발)   ← 5.3절
   6. 왼팔/오른팔 그룹 (피부 팔 — 몸통 "뒤")
   7. 몸통 + 머리 + 귀 + 볼터치 (core)
   8. 하의의 엉덩이 부분 (clip)  ← 몸통 아래단을 덮어야 하므로 core 뒤
   9. 상의 몸통 (소매는 추출되어 ↓로 이동)
  10. 소매 그룹 (좌/우)            ← 팔 스윙과 동기화
  11. 눈(.eyes 래퍼 — 깜빡임용) → 코 → 입
  12. 수염 → 얼굴 악세서리(안경 등) → 헤어 → 모자   ← 수염 위에 안경, 안경 위에 머리카락, 그 위에 모자
  13. 손(피부) + 손 아이템 그룹 (좌/우)  ← 맨 위: 검/우산이 모자·머리 위로 보여야 함
```

포인트:
- **배경/바닥장식은 #char 밖** → 캐릭터만 걷고 배경은 고정
- 렌더는 매번 전체 재생성(innerHTML). 이 규모(레이어 ~20개)에서는 성능 문제 전혀 없고,
  상태→화면 동기화 버그가 원천 차단됨
- 장착 직후 해당 레이어 `<g>`에 `.pop` 클래스 → scale 팝 애니메이션 (transform-box: fill-box)

## 5. 움직임(애니메이션) — 시행착오 포함 전부

### 5.1 1단계: 통짜 이동 (쉬움, 그러나 "종이인형" 소리 들음)

`#char` 그룹에 CSS 애니메이션만 걸면 착용 상태 그대로 움직인다:
- idle 둥실거림: translateY 진동
- 걷기: translateX 왕복 + 끝에서 `scaleX(-1)`로 방향 전환 (`transform-origin: 50% 100%`)
- `#char` 안에 `#char-inner`를 한 겹 더 두면 **이동(밖) + 바운스/기울임(안)** 두 애니메이션을 합성할 수 있다

실사용 피드백: "종이인형 같다" → 2단계로.

### 5.2 2단계: 관절 분리

몸을 그룹으로 쪼갠다: `다리L / 다리R / 팔L / 팔R / 몸통코어 / 손L / 손R`.
각 그룹을 관절 피벗 중심으로 회전:

```css
.limb { transform-box: view-box; }            /* transform-origin을 viewBox 좌표(px)로 지정하기 위해 필수 */
.leg-l { transform-origin: 146px 292px; }     /* 엉덩이 관절 */
.arm-l, .arm-l-mid, .arm-l-front { transform-origin: 135px 196px; }  /* 어깨 관절 */
```

**z-order 분리 + 동기화 트릭 (핵심):** 팔 피부는 몸통 "뒤", 손과 손 아이템은 상의 "위", 소매는 그 사이에
있어야 한다. SVG는 그리기 순서가 z-order라서 한 그룹으로 못 묶는다.
→ **같은 팔을 3개의 그룹(back/mid/front)으로 나눠 다른 z 위치에 두고, 셋에 동일한 keyframes·duration을
걸면 완벽히 동기화되어 한 팔처럼 움직인다.** (CSS 애니메이션은 같은 시점에 시작하면 어긋나지 않는다)

### 5.3 옷 입은 다리 움직이기 — clipPath 분할

하의/신발 아이템은 "양다리가 한 덩어리인 svg"라서 그대로는 다리별로 못 움직인다.
→ **clipPath로 좌/우/엉덩이 세 조각으로 잘라서, 다리 조각은 다리 그룹 안에, 엉덩이 조각은 몸통에 둔다:**

```html
<clipPath id="clip-hip"><rect x="0" y="0" width="320" height="280"/></clipPath>
<clipPath id="clip-leg-l"><rect x="0" y="270" width="160" height="130"/></clipPath>   <!-- x < 몸 중심선 -->
<clipPath id="clip-leg-r"><rect x="160" y="270" width="160" height="130"/></clipPath>
<!-- 같은 하의 아이템을 3번 렌더: 다리L그룹(clip-leg-l), 다리R그룹(clip-leg-r), 몸통(clip-hip) -->
```

여기서 얻은 교훈 3개 (전부 실제로 지적받고 고친 것):

1. **분할선은 무릎이 아니라 "허벅지 최상단(가랑이 바로 아래)"에 그어라.** 처음에 무릎 근처(y=302)에
   그었더니 긴바지만 움직이고 반바지·네모바지는 전부 분할선 위라 통째로 고정 → "하의가 그대로인데?" 피드백.
   y=270으로 올리니 해결
2. **이음새는 겹침으로 가린다.** 엉덩이 조각의 clip(…~280)을 다리 조각의 clip(270~…)보다 10px 더 내려
   겹치게 하고, 엉덩이 조각을 **나중에(위에)** 그린다. 다리가 회전해도 절단선이 겹침 띠 아래에 숨는다
3. **치마처럼 좌우로 쪼개면 어색한 아이템은 `rigid: true` 플래그**로 분할에서 제외하고 통짜로 몸통에 붙인다

### 5.4 소매 움직이기 — 마커 그룹 추출 패턴

상의도 한 덩어리 svg라 팔만 움직이면 "맨팔이 소매 밖에서 흔들리는" 어색함이 생긴다 (실제 피드백).
clipPath로 자르기엔 소매와 몸통이 겹쳐 직선으로 못 자른다. 해결책:

1. 데이터에서 소매를 `<g class="sleeve-l">…</g>` / `<g class="sleeve-r">…</g>`로 감싼다 (topBase 헬퍼가 자동으로)
2. 렌더러가 상의 svg 문자열에서 정규식으로 소매 그룹을 **추출·제거**:
   `svg.match(/<g class="sleeve-l">[\s\S]*?<\/g>/g)` (소매 안에 중첩 `<g>` 금지가 전제)
3. 추출한 소매를 팔 스윙 그룹(`arm-l-mid`)에 넣는다 → 팔 피부(뒤)·소매(중간)·손+아이템(앞)이 함께 회전

이 "마커 클래스 + 렌더 시 추출" 패턴은 범용적이다. "모자의 흔들리는 술 장식" 같은 것도 같은 방식으로 가능.

### 5.5 동작 시스템 — CSS 변수로 진폭 제어

동작(걷기/달리기/춤/인사/점프)을 추가할 때마다 keyframes를 새로 쓰지 않는다.
**keyframes는 공용 4~5개, 동작별 차이는 CSS 변수:**

```css
@keyframes swingA { 0%,100% { transform: rotate(var(--amp,13deg)); } 50% { transform: rotate(calc(-1 * var(--amp,13deg))); } }
@keyframes swingB { /* 반대 위상 */ }
@keyframes bob    { /* translateY(var(--bob)) + rotate(var(--lean)) — 바운스+상체 기울임 */ }

/* 동작 = 무대 svg에 붙는 클래스 하나 */
.action-walk .leg-l { --amp: 16deg; animation: swingA 0.76s cubic-bezier(.45,.05,.55,.95) infinite; }
.action-run  .leg-l { --amp: 27deg; animation: swingA 0.38s ease-in-out infinite; }
.action-run  #char-inner { --bob: -7px; --lean: 6deg; animation: bob 0.19s infinite; }
```

- 팔과 다리는 반대 위상(swingA/swingB) — 왼팔과 오른다리가 같은 위상 (실제 보행과 동일)
- "자연스러움"의 대부분은 **바운스 주기를 걸음의 1/2로 동기화**(걸음마다 한 번 출렁) + 상체 기울임에서 나옴
- 춤/인사 같은 비대칭 동작: 팔을 크게 드는 keyframes(armRaise)를 좌우 부호 반대로 만들고
  `animation-delay: -50%`로 번갈아 들게 함
- **연속 재생**: JS에서 `setInterval`로 동작 클래스를 4초마다 교체하는 10줄짜리 컨트롤러면 끝

### 5.6 잔재미 인터렉션 (체감 효과 큰 순)

1. **눈동자 마우스 추적**: pointermove → 얼굴 중심 기준 방향 벡터 → `.pupil`들에 `translate(±3px)` (rAF 스로틀)
2. **깜빡임**: `.eyes` 래퍼에 `transform: scaleY(0.08)` 클래스를 140ms 줬다 뗌, 다음 깜빡임은 2~6초 랜덤
3. **클릭 반응**: 점프 애니메이션 + 클릭 좌표에 떠오르는 "+1" 플로팅 라벨(HTML absolute, 0.9s 후 제거)

## 6. 상점/입어보기 UX (경제 기능을 쓸 경우)

- **카드 클릭 = 입어보기(무료 미리보기), 별도 구매 버튼 = 결제** — 클릭 즉시 구매시키면 바로 불만 나옴 (실제 피드백)
- 입어보기는 비저장 `preview` 상태 → 카드에 점선 테두리 + "입어보는 중" 배지, 탭/카테고리 이동 시 자동 해제
- 구매 즉시 자동 장착이 기분 좋다
- 인벤토리에도 상점과 같은 **카테고리 칩**을 달 것, 칩에 보유 개수 표시 (실제 피드백)
- 클릭 보상은 후하게 (클릭당 +10, 10% 확률 +100 수준) — 짜면 바로 "돈이 부족해" 소리 나옴

## 7. 팀위키 멤버 아바타로 이식할 때

1. **저장**: 멤버 프로필에 `{ face, equipped }` JSON 한 컬럼이면 끝. 아이템 카탈로그는 프론트 코드에 정적 포함
2. **렌더러를 순수 함수로 유지**: `renderAvatarSvg(avatarData, options) → string`.
   React/Vue면 `dangerouslySetInnerHTML`/`v-html`로 사용. 문자열 합성이라 SSR에서도 그대로 동작
3. **썸네일**: 같은 데이터로 `viewBox`를 얼굴 부근(예: `"90 50 140 140"`)으로 좁힌 작은 svg를 그리면
   "얼굴 클로즈업 프로필 사진"이 공짜로 나온다. 목록·댓글에는 클로즈업, 프로필 페이지에는 전신+배경+동작
4. **애니메이션은 옵트인**: 멤버 목록에 수십 명 렌더 시 전부 애니메이션이면 산만 + GPU 낭비.
   기본은 정지, 호버하거나 본인 프로필에서만 동작 재생 추천
5. **아이덴티티 보호**: 눈코입(얼굴)은 무료·자유, 꾸미기 아이템만 보상/포인트와 연동하면
   "포인트 없어서 얼굴도 못 만드는" 상황을 피할 수 있다
6. 슬롯 추천(위키 맥락): 헤어/모자/얼굴악세서리/상의/손 아이템/배경 정도로 시작, 수염·신발·바닥장식은 선택
7. 코스프레/세트 아이템(여러 슬롯을 모으면 캐릭터 완성)은 수집 동기 부여에 효과적 — 팀 이벤트 보상으로 좋음

## 8. 함정 목록 (전부 실제로 겪은 것)

| 함정 | 증상 | 해결 |
|---|---|---|
| 아이템 svg에 id/gradient 사용 | 썸네일 복제 시 id 충돌, 색이 엉뚱하게 나옴 | id·defs 금지, 단색+opacity |
| ES modules + `file://` | 더블클릭으로 열면 CORS로 전부 깨짐 | 모듈 없이 전역 스크립트 순서 로드 (또는 빌드 도구) |
| 분할선을 무릎에 그음 | 반바지·치마가 안 움직임 | 분할선은 가랑이 바로 아래로 |
| 분할 이음새 벌어짐 | 회전 시 흰 틈 | 정적 조각을 10px 겹쳐 위에 덮기 |
| 치마를 좌우 분할 | 천이 찢어진 듯 어색 | `rigid` 플래그로 통짜 처리 |
| 소매 고정 + 맨팔 스윙 | "옷은 가만히 있는데 관절만 움직임" | sleeve 마커 그룹 추출 패턴 |
| 마이그레이션 플래그를 병합 후 검사 | 보너스 지급이 영원히 안 됨 | 병합 전 원본(parsed)으로 판정 |
| transform-origin이 안 먹음 | 회전 중심이 엉뚱함 | SVG 요소엔 `transform-box: view-box`(관절) / `fill-box`(자기중심) 필수 |
| 한 그룹으로 팔 전체를 묶으려 함 | z-order가 깨짐 (팔이 몸통 위로) | back/mid/front 3그룹 + 동일 keyframes 동기화 |

## 9. 작업 순서 추천 (새 세션에서)

1. 몸 디자인 + 앵커 표 작성 (2.2) — 여기에 공들일 것, 이후 모든 작업의 기준
2. 레이어 순서 확정 + 렌더 함수 (4)
3. 눈코입 옵션 → 슬롯 1~2개에 아이템 3개씩만 먼저 (좌표 감 잡기) → SVG 검증 스크립트
4. 전체 카탈로그 양산 (헬퍼 함수 먼저 만들고)
5. 상태 저장/선택 UI
6. 인터렉션 (5.6) → 통짜 이동 (5.1) → 관절 (5.2~5.4) → 동작 시스템 (5.5)

각 단계마다 브라우저에서 실제 렌더를 확인하고 넘어갈 것. SVG 좌표 작업은 머릿속 예상과 실물이 자주 다르다.
